数据库常见概念

1. 数据库基础概念

  • 数据库（Database）
    一个存储数据的仓库，可以包含多个表，用于组织和管理大量结构化数据。

  • 表（Table）
    数据库中的一种数据结构，由行和列组成，用于存储特定类型的数据（如用户表、订单表）。

  • 列名（Column Name）
    表中的每一列都有一个名称，用于标识该列存储的数据（如 username、age）。

  • 数据类型（Data Type）
    定义列中数据的类型，如：

    • 整数类型：INT
    • 字符串类型：VARCHAR
    • 日期类型：DATE
    • 浮点数类型：FLOAT

  • 自增（Auto Increment）
    一种属性，通常用于主键列，每次插入新数据时自动递增（如 1, 2, 3…）。

  • 非空（NOT NULL）
    一种约束，确保列中不能存储空值（必须填写数据）。

  • 唯一（UNIQUE）
    一种约束，确保列中的值不能重复（但允许空值）。

  • 主键（Primary Key）
    一种唯一且非空的列（或列组合），用于唯一标识表中的每一行。

  • 外键（Foreign Key）
    一种列，用于关联另一张表的主键，确保数据的一致性和完整性。

  • 默认值（Default Value）
    如果插入数据时未指定值，列将自动填充的预设值（如 DEFAULT 0）。

  • 注释（Comment）
    对表、列或索引的说明，用于帮助理解其用途（如 COMMENT '用户姓名'）。

  • 索引（Index）
    一种数据结构，用于加快数据检索速度（类似于书的目录），但会增加写入数据的开销。

2. MySQL 存储引擎

  • MySQL 的默认存储引擎是 InnoDB。

  • 底层实现：使用 B+Tree 数据结构。

  • 支持事务（Transaction），四种事务隔离级别：

    1. 读未提交（Read Uncommitted）：允许事务读取未提交的数据，可能导致脏读。
    2. 读已提交（Read Committed）：只能读取已提交的数据，可能导致不可重复读。
    3. 可重复读（Repeatable Read）：默认隔离级别，防止不可重复读，但可能出现幻读。
    4. 串行化（Serializable）：最高级别的隔离，事务串行执行，保证数据一致性，但性能最低。

  • 支持行级锁和表级锁，可以提高并发访问能力。

  • 具有崩溃恢复能力，在系统故障后可恢复数据。

SQL 数据类型选择指南

在 SQL 中，选择合适的数据类型至关重要，它影响 存储效率、查询性能 以及 数据完整性。以下是常见的数据类型分类及其使用建议。

1. 数值类型（Numeric Types）

数值类型主要用于存储整数或小数，适用于 ID、计数、价格等数据。

数据类型 大小 适用场景
TINYINT 1 字节 适用于 小范围整数（0255 或 -128127）
SMALLINT 2 字节 适用于 中等范围整数（-32,768~32,767）
MEDIUMINT 3 字节 适用于 较大范围整数（-8,388,608~8,388,607）
INT / INTEGER 4 字节 常用整数类型（-2,147,483,648 ~ 2,147,483,647）
BIGINT 8 字节 超大整数，适用于 存储社交媒体用户 ID、计数器等
DECIMAL(m,n) 变长 精准存储小数，适用于 货币、财务计算（避免浮点数误差）
FLOAT 4 字节 浮点数，适用于 科学计算（小数位数有限）
DOUBLE 8 字节 双精度浮点数，比 FLOAT 更高精度

✅ 选择建议：

  • ID、计数器 用 INT 或 BIGINT（大规模数据）。
  • 价格、金额 用 DECIMAL(10,2)，避免浮点数精度问题。
  • 科学计算 用 FLOAT 或 DOUBLE，但不适合财务数据。

2. 字符串类型（String Types）

用于存储文本信息，如姓名、地址、描述等。

数据类型 大小 适用场景
CHAR(n) 固定长度（最大 255） 适用于 固定长度字符串（如性别、状态码、国家代码）
VARCHAR(n) 变长（最大 65535） 最常用，适用于 可变长度字符串（如用户名、电子邮件）
TEXT 变长（最大 4GB） 大文本（如文章内容、日志）
BLOB 变长（最大 4GB） 二进制数据（如图片、音频、视频）

✅ 选择建议：

  • 短字符串（固定长度） 用 CHAR（存储效率高）。
  • 普通文本（可变长度） 用 VARCHAR（占用空间少）。
  • 超长文本 用 TEXT，但 TEXT 不支持索引！
  • 存储文件/图片 用 BLOB，但建议用路径存储文件名，数据存放于文件系统。

3. 日期和时间类型（Date & Time Types）

用于存储日期、时间和时间戳。

数据类型 大小 适用场景
DATE 3 字节 仅存储 年月日（YYYY-MM-DD）
TIME 3 字节 仅存储 时分秒（HH:MM:SS）
DATETIME 8 字节 常用，存储完整时间（YYYY-MM-DD HH:MM:SS）
TIMESTAMP 4 字节 存储 Unix 时间戳（1970-2038）
YEAR 1 字节 存储 年份（YYYY，1901-2155）

✅ 选择建议：

  • 仅存储日期 用 DATE。
  • 存储完整时间 用 DATETIME（更通用）。
  • 存储时间戳 用 TIMESTAMP（可自动更新）。
  • 存储年份 用 YEAR（节省空间）。

4. JSON 和其他类型

数据类型 适用场景
JSON 存储 JSON 格式 数据，适用于 灵活数据存储
ENUM 限制值的范围（如性别 ('Male', 'Female')）
SET 可选多个值的集合

✅ 选择建议：

  • 存储 JSON 数据 用 JSON（但查询性能较差）。
  • 限制值范围（状态、性别） 用 ENUM。
  • 多选字段（多个标签） 用 SET。

5. 数据类型选择总结

  • 整数 ID 用 INT 或 BIGINT，根据数据量选择。
  • 浮点计算（科学计算） 用 FLOAT 或 DOUBLE，但不适用于金额。
  • 货币金额 用 DECIMAL，避免浮点误差。
  • 短文本（20 字符以内） 用 CHAR，固定长度存储更快。
  • 普通字符串 用 VARCHAR，节省存储空间。
  • 长文本 用 TEXT，但 TEXT 不支持索引。
  • 日期时间 用 DATETIME，或者 TIMESTAMP（自动更新时间）。

6. 数据类型优化建议

  1. 尽量使用合适的最小存储类型：

    • TINYINT（1 字节）比 INT（4 字节）更节省空间。
    • VARCHAR(50) 比 VARCHAR(255) 速度更快。

  2. 避免不必要的 TEXT 和 BLOB：

    • TEXT/BLOB 无法建立索引，查询效率低。
    • 建议使用 VARCHAR 存储小型文本。

  3. 索引列避免使用 VARCHAR：

    • INT 作为索引比 VARCHAR 更高效。
    • 尽量使用 数值 ID 作为主键，不要用 UUID/VARCHAR。

总结

  • 节省存储空间，使用合适的数据类型。
  • 优化查询性能，避免 TEXT/BLOB 影响索引。
  • 索引字段尽量用数值类型，VARCHAR 适用于非索引字段。
  • 使用 DECIMAL 存储金额，避免 FLOAT/DOUBLE 精度丢失。

正确的数据类型选择能大幅提升 SQL 的存储效率和查询速度！
